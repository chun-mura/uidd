# uidd v0.7.0 — vrt / token-lint コマンド + sb-verify 状態網羅チェック 設計

日付: 2026-07-19
ステータス: 設計承認済み・実装前

## 目的

一般的なデザインプロセスと uidd 現行資産のギャップ分析 (2026-07-19) で優先度高とした3項目を実装する:

1. **vrt コマンド**: ビジュアルリグレッション検証。マスター設計書がフェーズ2予告していた VRT を、ツール検出 + ローカル基準のハイブリッドで案件インフラ非依存に実現する
2. **token-lint コマンド**: トークン準拠の継続検査。マスター設計書の対象業務6「デザイントークンの一元管理・変換」の未着手部分。figma-implement のマッピング工程 (一回性) を継続検査化する
3. **sb-verify チェック4「状態網羅」**: コンポーネント単位のストーリー網羅 (チェック1) では見えない、状態単位の穴を検出する

あわせて、同ギャップ分析の優先度中・低の候補をマスター設計書に「将来候補」として記録する (本設計のスコープ外、記録のみ)。

## 既存資産との棲み分け

| 資産 | 役割 | 新規資産との関係 |
|---|---|---|
| `sb-verify` | a11y・インタラクション・網羅の機械検証 | vrt は見た目の退行検知で別関心事 (sb-verify に統合しない: 実行コスト・baseline 管理が異質)。チェック4 は sb-verify 自体の拡張 |
| `ds-audit` | ゼロからの棚卸し (色・spacing・重複コンポーネント、一回性・広域) | token-lint はトークン準拠の継続検査 (狭く・繰り返し)。両ファイルに相互ポインタを明記し重複を作らない |
| `figma-implement` | 実装時のトークンマッピング (一回性) | token-lint は同じマッピング判断を既存コード全体への継続検査として提供 |
| `ui-propose` | キャプチャ環境の依存検出パターンの出所 | vrt が同じ判定 (ブラウザ MCP + storybook スクリプト) を流用する |

## コンポーネント1: `commands/vrt.md`

入力: 対象ストーリー (省略時全件) → 出力: `docs/vrt/YYYY-MM-DD-vrt.md` (対象プロジェクトのルートからの相対パス。PASS/FAIL ブロック形式は sb-verify と同形式)

### 依存検出

1. **VRT ツール検出**: 対象プロジェクトの package.json に Chromatic (`chromatic`) または reg-suit (`reg-suit`) があれば、そのツールの実行をオーケスト レーションし結果を要約する (ローカル基準と二重管理しない)。両方あれば AskUserQuestion でどちらを使うか確認する
2. ツールが無ければ**ローカル基準モード**: キャプチャ環境を検出する (ui-propose と同じ判定: (a) ブラウザ操作系 MCP、(b) package.json の storybook 起動スクリプト)。**どちらか欠ければ中断する** — VRT はキャプチャなしで縮退できないため、欠けている依存とセットアップの選択肢 (ブラウザ MCP 導入 / Chromatic・reg-suit 導入) を案内して止める。無言のフォールバック禁止

### ローカル基準モードのフロー

1. **baseline 確認**: `docs/vrt/baselines/<story-id>.png` (対象プロジェクトのルートからの相対パス) の有無をストーリーごとに確認する
2. **初回 baseline 作成**: baseline が無いストーリーは Storybook をローカル起動してキャプチャし、baseline として保存する。このストーリーは「baseline 作成 (比較なし)」として報告する
3. **比較**: baseline があるストーリーは現状をキャプチャし比較する。画像 diff ツール (ImageMagick `compare` / `pixelmatch` 等) が使えれば数値差分を取り、使えなければ「目視比較モード」と明示して両画像を並べて比較する
4. **判定**: 差分ありは FAIL とし、レポートに baseline と現状の両画像パスを記載する。差分なしは PASS
5. **baseline 更新**: FAIL 分について意図した変更か退行かはツールに判断できないため、**必ず AskUserQuestion でユーザー承認を得てから** baseline を更新する。承認なしで上書きしない
6. **報告**: `docs/vrt/YYYY-MM-DD-vrt.md` に PASS/FAIL/baseline 作成の3区分で保存する。起動したプロセスは検証後に停止する

## コンポーネント2: `commands/token-lint.md`

入力: 対象ディレクトリ → 出力: `docs/audit/YYYY-MM-DD-token-lint.md` (対象プロジェクトのルートからの相対パス)

### 依存検出

1. 引数が空なら AskUserQuestion で対象ディレクトリを確認する。パスが存在しなければ「対象ディレクトリが見つからない: <パス>」と明示して中断する。推測で続行しない
2. **トークン定義の検出**: CSS variables / Tailwind config / style-dictionary / theme ファイル等を探索する。見つからなければ AskUserQuestion でパスを確認し、それでも無ければ「トークン定義なし: ハードコード値の列挙のみ行う」と明示縮退する
3. **Figma MCP 検出**: あれば variables を取得しコード側トークンとの乖離 diff を報告する。無ければ「Figma 差分なしで続行する」と明示縮退する

### フロー

1. **トークン定義読込**: 検出した定義から色・spacing・タイポグラフィ等のトークン一覧を作る
2. **ハードコード値検出**: 対象コードから生値 (色 hex/rgb・spacing px 値・font-size 等) を検出する
3. **最近傍トークン提案**: 各検出値に最も近い既存トークンを提案する。近いトークンが無い値は「トークン化候補」として別掲する
4. **Figma 乖離 diff** (MCP がある場合のみ): コード側トークンと Figma variables を突き合わせ、値の乖離・片側にしか無い定義を一覧化する
5. **報告**: `docs/audit/YYYY-MM-DD-token-lint.md` に保存する

### ds-audit との棲み分け (両ファイルに明記)

ds-audit = ゼロからの棚卸し (重複コンポーネント含む広域・一回性)。token-lint = トークン準拠の継続検査 (狭く・繰り返し使う)。

## コンポーネント3: `sb-verify.md` チェック4「状態網羅」

- コンポーネントの props・条件分岐から表現可能な UI 状態 (empty / loading / error / disabled 等) を推定し、ストーリー未定義の状態を検出する
- **props/分岐に存在するのにストーリーが無い状態は FAIL**
- hover / focus 等の疑似クラス状態は参考情報として列挙のみ (ストーリー化必須としない。FAIL の根拠にしない)
- 既存の3チェックと同じ独立 PASS/FAIL ブロックで報告し、他チェックの結果に影響しない
