# uidd — UI-Driven Development Toolkit 設計

日付: 2026-07-12
ステータス: フェーズ1 実装済み (v0.2.0 で初期資産、v0.3.0 でクライアント配布対応)

## 目的

Figma の代わりに Storybook を使ってクライアント・社内に UI 提案を行う業務を、Claude Code で仕組み化する。対象業務:

1. デザインシステムの構築・レビュー・検証
2. UI パーツの作成・更新・レビュー・検証
3. Storybook の作成・更新・レビュー・検証
4. (精査で追加) 提案成果物の生成・共有フロー
5. (精査で追加) UI 判断ルールのコード化
6. (精査で追加) デザイントークンの一元管理・変換
7. (精査で追加) 既存フロントエンドからのデザインシステム抽出 (監査)

参考事例: [NEWT の AI 駆動デザインシステム](https://note.com/toitoi1618/n/ndf35dbd2585b) — デザインシステムのシングルソース化、design-builder スキル + 判断ルール集、調査・複数案・仕様を1枚にまとめた提案成果物、プレビュー URL での即時共有。

## 前提 (確定事項)

- 対象範囲: 複数のクライアント案件で横断利用 (案件非依存に設計する)
- 利用者: 本人 → 社内 → 将来的にクライアントにも提供 (配布可能な形式・ドキュメントが必要)
- 技術スタック: React + TypeScript と Vue / Nuxt の両方 (フレームワーク非依存を基本とし、コード生成部分のみ分岐)
- 既存資産: ゼロから構築する案件と、既存フロントエンドを元に構築する案件の両方に対応する

## aidd / superpowers との棲み分け (三層構造)

```
superpowers  … どう進めるか (process): brainstorming, TDD, デバッグ, 計画
aidd         … 何をさせるか (汎用 task): design-review, test-perspectives, ADR, reviewer
uidd         … 何をさせるか (UI ドメイン特化): 判断ルール, 提案フロー, Storybook 検証
```

- aidd はチーム共有・公開に対応済み (v0.13.0〜) だが、それでも別リポジトリとする。根拠: (1) ドメインが異なる — aidd は汎用開発タスク、uidd は UI/デザイン特化。統合するとインデックスが肥大化し「1ファイル1関心事」の運用が崩れる。(2) ライフサイクルが異なる — uidd は Storybook エコシステム・UI 案件とともに進化する。(3) 導入単位を分けられる — バックエンドのみのプロジェクトは aidd だけ導入すればよい
- uidd は aidd の資産 (design-review / test-perspectives / reviewer) を再利用し、重複する機能を作らない。uidd 側の docs にポインタを書く
- **aidd への依存はポインタ参照のみ**: uidd の commands/skills は aidd を実行時に呼ばず単体で完結する (aidd 未導入環境でも動作する)。将来実行時に呼ぶ資産を作る場合は、未導入検出 + 明示エラーでの中断を必須とする
- aidd の README には uidd へのポインタを1行追加する (aidd 運用ルール4「重複禁止・ポインタのみ」に準拠)
- superpowers のスキルと被る process 系機能は作らない

判断基準は aidd と同じ: 「どの案件・フェーズでも同じ動きをするか」→ Yes なら superpowers/aidd を使う。「UI・デザインという成果物ドメインに紐づくか」→ Yes なら uidd。

## 形式: プラグイン + テンプレートリポジトリの2点構成

| 形式 | 役割 | 時期 |
|---|---|---|
| **uidd プラグイン** (本リポジトリ) | 案件非依存のワークフロー資産: skills / commands / agents。marketplace 経由で配布 | フェーズ1 (今) |
| **テンプレートリポジトリ** (別リポジトリ、名称未定) | 案件ごとに複製するデザインシステム雛形: デザイントークン管理 + Storybook 設定 + 公式 MCP アドオン + uidd 導入済み設定 (ツール選定等の詳細はフェーズ2着手時に決定) | フェーズ2 |

**MCP サーバは自作しない**: Storybook 公式が `@storybook/addon-mcp` を提供している (コンポーネント一覧・props 取得・ストーリー生成指示・テスト実行)。現状 React 系限定のため、Vue 案件ではファイル直読みにフォールバックする。参照: https://storybook.js.org/docs/ai

## ディレクトリ構成

```
uidd/
├── .claude-plugin/
│   ├── plugin.json          # name: uidd
│   └── marketplace.json     # marketplace 定義 (uidd)
├── skills/                  # UI 判断ルール等 (自動トリガする知見)
├── commands/                # /uidd:* コマンド
├── agents/                  # UI 特化サブエージェント (必要になったら)
├── docs/
│   ├── tips/                # 方法論の知見・aidd/superpowers へのポインタ (フェーズ2 で必要になったら作成。現状なし)
│   └── superpowers/specs/   # 設計ドキュメント (本ファイル含む)
└── README.md                # 全資産インデックス + 導入手順
```

## 初期コンテンツ (最小セット)

aidd と同じく「最初から網羅しない」。実案件で2回以上使ったものだけ昇格させる (2回ルール)。フェーズ1 の骨格:

| 資産 | 形態 | 入力 → 出力 | 内容 |
|---|---|---|---|
| `commands/ui-propose.md` | command | 対象画面/要件資料のパス → `docs/proposals/<日付>-<名前>.md` | 提案フロー: 前提確認 → 資料読込 → 調査 → 複数案実装 → 理由・仕様付き提案成果物の生成 |
| `commands/ds-audit.md` | command | 対象ディレクトリ → `docs/audit/<日付>-audit.md` | 既存フロントエンドの棚卸し: 色・spacing・重複コンポーネントを抽出し統合候補を提案 |
| `commands/sb-verify.md` | command | 対象ストーリー (省略時全件) → 検証項目ごとに独立した PASS/FAIL ブロック | Storybook 検証の**オーケストレーター**: ストーリー網羅チェック / a11y / インタラクションテスト / 状態網羅の4チェックを順に呼ぶ。各チェックは独立しており、結果も項目別に報告する |
| `skills/ui-judgment/` | skill | (自動トリガ) | UI 判断ルール集 (余白・階層・状態設計・命名)。実案件での判断を都度追記して育てる |

検証系 (VRT: Chromatic / reg-suit) は v0.7.0 で `commands/vrt.md` としてコマンド化済み (ツール検出 + ローカル基準のハイブリッド。詳細は 2026-07-19-vrt-token-lint-state-coverage-design.md)。**提案成果物の「共有」(プレビュー URL の発行・配布) も同様にホスティングが案件インフラに依存するため、フェーズ1 では docs/tips に手順のみ書き、仕組み化はフェーズ2 のテンプレートリポジトリ側の責務とする** (フェーズ1 の `ui-propose` は成果物の生成まで)。

## 運用ルール (aidd 踏襲)

1. 1ファイル1関心事。docs は冒頭に「いつ使うか」を明記 (sb-verify はオーケストレーターとして例外的に4チェック (状態網羅含む) を束ねるが、各チェックの定義と結果報告は独立させる)
2. 2回ルール: 2回以上使ったプロンプトだけ commands/skills に昇格。カウントは当面本人の利用実績のみを基準とし、社内展開後の利用実態の収集方法は展開時に定める
3. README がインデックス: 資産追加時に更新し、plugin.json の version を上げる
4. 重複禁止: superpowers / aidd と被る機能は作らずポインタのみ
5. 受動ドキュメント禁止: セッション中に効かせたい知見は skills か hooks にする
6. 判断ルール (skills/ui-judgment) は案件での実判断を根拠として追記する。根拠のない一般論を書かない。**案件セッション中に得た判断ルールは必ず uidd 本体に還元してから各案件へ反映する (uidd 本体が単一の真実源。案件側ローカルでの独自追記を正としない)**
7. ロールバック: 配布後に問題が出たバージョンは該当コミットへ revert し、plugin.json の version を上げ直して再配布する
8. 社内展開: aidd v0.13.0 が確立した方式を踏襲する。利用プロジェクトの `.claude/settings.json` に team-settings テンプレートをマージしてコミットし、プロジェクトを trust したメンバーへ自動でインストール提案する (各自の `/plugin marketplace add` は不要)。更新配信は plugin.json の version に固定されるため、リリース時は必ず version を上げて CHANGELOG に記録する (運用ルール3)。メンバーは `/plugin update uidd` で追従する

## エラーハンドリング / テスト

- プラグインとして読み込めることを `/plugin` 経由で検証してから完了とする
- commands は最低1回実案件で動作確認してから README に掲載する
- **依存検出と degrade 契約**: 各 command は実行冒頭で依存を検出し、不成立時に silent に続行しない
  - `@storybook/addon-mcp` の有無を検出し、無ければ「ファイル直読みモードで続行する」と明示出力してから処理する (Vue / Nuxt 案件は常にこの経路)
  - aidd 資産は実行時に呼ばない (棲み分けセクション参照)。呼ぶ資産を将来作る場合は未導入検出 + 明示エラーで中断する

## スコープ外

- MCP サーバの自作 (公式 `@storybook/addon-mcp` を使う)
- テンプレートリポジトリの実装 (フェーズ2。本設計では位置づけのみ定義)
- Figma 連携 (Storybook 中心のワークフローのため)
- クライアント向け公開の具体的な配布手順 (フェーズ1 はローカル marketplace のみ。公開時に README へ追記)

### 優先度中

- **ヒューリスティック UI レビュー command**: uiux-principles + ui-judgment を評価軸に、ストーリー/キャプチャを対象とした原則ベースのレビュー成果物を出す。機械検証の sb-verify と対になる (aidd:design-review は汎用設計レビューのため棲み分け可能)
- **ハードコード値検出 hook**: 実装中に生値 (色 hex 等) を書いた時点で既存トークンを提案する受動的強制。運用ルール5 に整合。uidd 初の hook になる (aidd に前例あり)。token-lint (事後検査) の実行時版
- **提案成果物の共有 (プレビュー URL 発行・配布)**: フェーズ2 テンプレートリポジトリ側の責務 (本設計書「初期コンテンツ」節で定義済み)。テンプレートリポジトリ着手時に実装する

### 優先度低

- **figma-implement の画面単位対応**: 現在は意図的にコンポーネント単位のみ。積み上げ運用で不足が実際に出てから
- **Code to Canvas (実装 → Figma 書き戻し)**: デザイナーレビューを Figma 上で受けるフローには有効。需要が発生してから
- **UX ライティング/マイクロコピー検査**: まず uiux-principles の1節として薄く足すのが先。独立資産は2回ルール待ち
- **ユーザビリティテスト計画支援**: 上流は superpowers (brainstorming) / reqd と重なるため、置くなら「Storybook を被験材料にしたテスト計画」に限定する
