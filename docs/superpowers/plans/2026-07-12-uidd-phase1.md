# uidd フェーズ1 (初期コンテンツ4点) 実装計画

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** uidd プラグインのフェーズ1 資産4点 (ui-propose / ds-audit / sb-verify コマンドと ui-judgment スキル) を実装し、`/plugin` 経由で読み込めることを検証する。

**Architecture:** プラグインは markdown 資産のみで構成する (実行コードなし)。commands は aidd の command 形式 (YAML frontmatter `description` + `argument-hint`、本文は日本語の指示プロンプト) を踏襲し、各コマンド冒頭に依存検出と degrade 契約を明記する。skill は aidd の SKILL.md 形式 (frontmatter `name` + 英語 `description`、本文日本語) を踏襲する。

**Tech Stack:** Claude Code プラグイン (markdown commands / skills)、`claude plugin` CLI (validate / marketplace / install / details)。

**Spec:** `docs/superpowers/specs/2026-07-12-uidd-design.md` (承認済み・レビュー反映済み)

## Global Constraints

- 依存検出と degrade 契約 (設計書「エラーハンドリング/テスト」節): 各 command は実行冒頭で依存を検出し、不成立時に silent に続行しない。`@storybook/addon-mcp` が無ければ「ファイル直読みモードで続行する」と明示出力してから処理する (Vue / Nuxt 案件は常にこの経路)
- aidd 資産は実行時に呼ばない: uidd の commands/skills は単体で完結する (aidd 未導入環境でも動作する)。aidd への言及はポインタのみ
- 運用ルール1: 1ファイル1関心事。sb-verify のみオーケストレーターとして例外的に3チェックを束ねるが、各チェックの定義と結果報告は独立させる
- 運用ルール3: 資産追加時に README インデックスを更新し、plugin.json の version を上げ、CHANGELOG に記録する (今回: 0.1.0 → 0.2.0)
- 運用ルール6: ui-judgment のルールは実案件での実判断のみを根拠とする。根拠のない一般論を書かない (初期状態はカテゴリ枠のみで空)
- README 掲載方針 (ユーザー確認済み): commands 3点は「実案件未検証」の注記付きで掲載する。実案件で1回動作確認したら注記を外す (設計書エラーハンドリング節との折衷)
- 完了条件: `/plugin` 経由で読み込めることの検証まで (設計書エラーハンドリング節)
- スコープ外 (ユーザー確認済み): docs/tips (VRT 手順・提案共有手順) は今回作らない。テンプレートリポジトリ (フェーズ2) に着手しない
- 言語: 資産本文・会話は日本語、commit メッセージは英語。commit には変更の「なぜ」を書く
- 各タスク完了ごとに `claude plugin validate .` が exit 0 (`✔ Validation passed`) であることを確認してから commit する

1. 対象ディレクトリが存在しない、またはフロントエンドコード (コンポーネント・スタイル) が見つからない場合は「監査対象が見つからない: <パス>」と明示して中断する。推測で続行しない

3. **重複コンポーネント**: 同じ役割のコンポーネント (Button 系・Modal 系・Input 系等) を横断で列挙し、props と見た目の差分を比較する

`commands/sb-verify.md` を以下の内容で作成する:

````markdown
---
description: Storybook のストーリー網羅・a11y・インタラクションテストを検証するオーケストレーター

## 依存検出 (処理前に必ず行う)

1. Storybook 未導入 (`.storybook/` が無く package.json にも storybook が無い) なら「Storybook が導入されていない」と明示して中断する
2. Storybook MCP の有無を検出する: セッションに Storybook 系 MCP ツールが見えるか、package.json に `@storybook/addon-mcp` があるかを確認する。無ければ「ファイル直読みモードで続行する」と明示出力してから処理する (Vue / Nuxt 案件は常にこの経路)
3. テスト実行環境 (`@storybook/test-runner` 等) の有無を確認する。無ければチェック2・3 は静的検査に degrade する (各チェックの定義参照)

## オーケストレーション

以下の3チェックを順に実行する。各チェックは独立しており、あるチェックの FAIL や実行不能が他チェックの実行を止めない。結果は必ずチェックごとに独立した PASS/FAIL ブロックで報告する (まとめて1つの判定にしない)。

### チェック1: ストーリー網羅

対象コンポーネントごとに検査する: ストーリーが存在するか。コンポーネントが持つ主要な状態 (variant 系 props、disabled、loading、error、empty 等、実装から読み取れるもの) がストーリー化されているか。
判定: 全コンポーネントにストーリーがあり主要状態がカバーされていれば PASS。不足があれば FAIL とし、不足しているコンポーネント・状態を列挙する。

### チェック2: a11y

テスト実行環境があれば a11y チェック (test-runner + axe) を実行し violation を集計する。無ければ静的検査に degrade し、その旨を明示する: img の alt、ボタン・リンクのアクセシブルネーム、フォームコントロールと label の紐付け、コントラストが疑わしい配色をコードから列挙する。
判定: violation 0 なら PASS (静的検査時は「PASS (静的検査のみ)」と明示)。あれば FAIL とし、件数と内訳を報告する。

### チェック3: インタラクションテスト

play function を持つストーリーを列挙し、テスト実行環境があれば実行する。無ければ「実行環境なし。play function の有無のみ検査する」と明示する。インタラクションを持つコンポーネント (フォーム・開閉・選択等) で play function が無いものを列挙する。
判定: 実行して全通過なら PASS (静的検査時は「PASS (静的検査のみ)」)。実行失敗、またはインタラクションがあるのに play function が無ければ FAIL。

## 報告形式

```
## sb-verify 結果 (対象: <対象>)

### チェック1: ストーリー網羅 — PASS / FAIL
<詳細>

### チェック2: a11y — PASS / PASS (静的検査のみ) / FAIL
<詳細>

### チェック3: インタラクションテスト — PASS / PASS (静的検査のみ) / FAIL
<詳細>
```
````
