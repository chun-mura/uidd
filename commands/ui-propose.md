対象: $ARGUMENTS

## 依存検出 (処理前に必ず行う)

1. `$ARGUMENTS` が空なら AskUserQuestion で対象画面・要件資料のパスを確認する。パスが存在しなければ「対象資料が見つからない: <パス>」と明示して中断する。推測で続行しない
2. Storybook MCP の有無を検出する: セッションに Storybook 系 MCP ツールが見えるか、対象プロジェクトの package.json に `@storybook/addon-mcp` があるかを確認する。無ければ「ファイル直読みモードで続行する」と明示出力してから処理する (Vue / Nuxt 案件は常にこの経路)
3. キャプチャ環境の有無を検出する: (a) セッションにブラウザ操作系 MCP ツール (Chrome MCP 等) が見えるか、(b) 対象プロジェクトの package.json に storybook 起動スクリプトがあるか。両方あればレンダリング検証 (フロー5) を行う。どちらか欠ければ「キャプチャなしで続行する」と明示出力し、フロー5 をスキップして成果物に「キャプチャ: 未実施 (環境なし)」と記載する

## フロー

1. **前提確認**: 資料から以下が読み取れない場合、実装前に AskUserQuestion で確認する: 提案の目的 (新規/改善)、対象ユーザー、制約 (既存デザインシステム準拠か・対応ブレークポイント)、案の数 (デフォルト 2〜3案)
2. **資料読込**: 対象資料と、プロジェクトの既存デザイントークン・共通コンポーネントを読む。MCP があればコンポーネント一覧・props を取得し、無ければコンポーネントディレクトリと既存ストーリーを直読みする
3. **調査**: プロジェクト内の類似 UI と一般的なパターンを調査し、案の比較軸 (操作コスト・実装コスト・拡張性・a11y) を決める
4. **複数案実装**: 各案を対象プロジェクトのフレームワークに合わせてコード + ストーリーとして実装する。案ごとに独立して動作する状態にする。UI 上の判断 (余白・階層・状態設計・命名) は skills/ui-judgment のルールに従い、ルールにない新しい判断をしたら ui-judgment の昇格候補への記録を提案する (2回ルール。詳細はスキル参照)
5. **レンダリング検証** (キャプチャ環境がある場合のみ): Storybook をローカル起動し、各案のストーリーを実際に表示してスクリーンショットを撮る。保存先は対象プロジェクトのルートからの相対パス `docs/proposals/assets/YYYY-MM-DD-<slug>/<案名>.png`。表示崩れ・意図と異なるレンダリングがあれば該当案のコードを修正し、再キャプチャしてから次へ進む。起動したプロセスは検証後に停止する
6. **提案成果物生成**: 対象プロジェクトのルートからの相対パス `docs/proposals/YYYY-MM-DD-<slug>.md` に保存する (日付は `date +%F`、slug は対象の英語ケバブケース要約)。ディレクトリがなければ作成する。キャプチャがあれば各案のセクションに相対パスで埋め込む。保存後、パスを報告する

## 成果物の構成

1. 前提 (目的・制約・対象ユーザー)
2. 調査まとめ (既存資産・参考パターン・比較軸)
3. 各案: ストーリー参照 (ストーリー ID or ファイルパス) / キャプチャ画像 (未実施なら「キャプチャ: 未実施 (環境なし)」と明記) / 設計理由 / 仕様 (props・状態・レスポンシブ挙動) / トレードオフ
4. 推奨案と理由

**スコープ**: 成果物の生成まで。プレビュー URL の発行・配布 (共有) はホスティングが案件インフラに依存するためフェーズ1 のスコープ外 (設計書参照)。
